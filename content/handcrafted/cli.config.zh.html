先讲人话版

🔧 openclaw config 像“一把专门拧配置螺丝的小扳手”

如果 configure 是装修向导，那 config 就是你自己下场动手时用的小扳手。它不陪你聊天，它直接对着 openclaw.json 的具体位置动手。

你可以把它想成“精确修改器”。想看某一颗螺丝在哪，想换一颗，想把一颗卸掉，想整体验证有没有拧歪，都是它的活。

适合不想走向导、只想精确改某个配置点的人 这页最重要的是分清：get/set/unset 在动螺丝，file/schema/validate 在看图纸 查看官方原文

第一站

🧰 先把它分成两类动作：动配置，和看图纸

这页命令很多，但其实很好分。

🔍

get

像拿手电照一下某颗螺丝现在拧在哪。

🔩

set

像把某颗螺丝拧进新位置，或者换成另一种零件。

🪛

unset

像把某颗螺丝卸下来，让那里恢复成“没装这个”的状态。

📐

file / schema / validate

像看图纸放在哪、标准长什么样、现在有没有装歪。

第二站

📍 路径就是“哪一颗螺丝”，值就是“你要拧成什么样”

看起来像长长的点号路径，其实就是在说“具体改哪一处”。

openclaw config get browser.executablePath

像先问：“浏览器可执行文件这颗螺丝，现在指向哪？”

openclaw config set gateway.port 19001

像直接把门牌号从一个数拧到另一个数。

agents.list[0].tools.exec.node

像不只拧大厅螺丝，还能钻进第一个 agent 的小抽屉里拧里面那颗。

unset ...apiKey

像把一颗凭证螺丝整个拆掉，不留假的空壳。

第三站

🪪 SecretRef builder 和 provider builder，像“帮你装钥匙柜”和“帮你做取钥匙机器”

这是这页最容易把人看晕的地方，但一旦换成画面就很好懂。

🔐 SecretRef builder

像告诉系统：“这把钥匙不放明面上，它在某个钥匙柜里，去那里拿。”

🏭 provider builder

像不只是指向钥匙柜，而是直接装一台“取钥匙机器”，比如从 env、文件、命令里拿。

--ref-provider --ref-source --ref-id

像给钥匙柜写上“去哪拿、谁负责、编号几号”。

--provider-command ...

像你不是从柜子拿，而是让一台取件机去执行一段命令，把钥匙取回来。

第四站

📦 Batch mode 是“别一颗颗拧，直接拿一张施工单一起改”

当你一次要改很多处时，batch 就像一张集中施工单。

🧾

--batch-json

像你把一整张施工清单直接贴在命令后面。

📄

--batch-file

像你把施工单另外写在一个文件里，再让它按单子施工。

📌

批处理以单子为准

官方特地说 batch payload 才是唯一真相，不会再被别的解析模式扰乱。

🎈

一句话

改一堆时，别一颗颗拧，直接交施工单。

第五站

🧪 --dry-run 很重要，它像“先拿样板比一下，不真下手”

这页最稳的地方就是 dry run。尤其改 SecretRef、provider、批量配置时，先试演一遍特别值。

--dry-run

像装修师傅先在墙上比划，不立刻打孔。

schema checks

像先看你这颗零件形状对不对，别硬塞错口。

resolvability checks

像再确认那把钥匙柜里真的有钥匙，不是写了编号却拿不到。

--allow-exec

像你愿意让“取钥匙机器”也真的先跑一遍，看看它能不能把钥匙带回来。

第六站

✅ validate 是“整屋巡一圈，看看有没有哪颗螺丝已经拧歪”

这和 set 不一样。它不改，只检查。

openclaw config validate

像叫验收员进屋，从头到尾看一遍现在的配置有没有违反图纸。

--json

像验收报告不要口头念，直接给机器能抄走的结构化单子。

config file / config schema

一个像告诉你图纸放在哪个抽屉，一个像把设计标准本身摊开给你看。

🎈 一句话

改完配置后，拿它做一次整屋验收特别值。

最后总结

🎈 把 openclaw config 这页压成一句话

openclaw config 就是一把精细扳手：你可以点对点地看、拧、拆配置，也可以用施工单批量改，再用 dry run 和 validate 先试演、再验收。

如果你下一页只想继续看一页，我建议看 /cli/update。因为房子装好以后，下一个常见问题就是：怎么安全升级，不把屋子搞塌。
