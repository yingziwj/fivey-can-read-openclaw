先讲人话版

Publishing 就是“把自己的本领盒子正式摆上 ClawHub 货架”

发布不是把文件随手丢上网。ClawHub 会看 owner、名字、版本、文件、范围和权限，确认这盒东西该由谁摆上架。

如果说安装是顾客拿货，发布就是店主交货：要贴对标签，也要确认你真的有权把它放到这个位置。

适合准备发布 skill 或 plugin 的维护者 重点是 owner、包名、版本和发布流程 查看官方原文

第一站

owner 像货架上的店名，包名要和店名对得上

ClawHub 里的 owner 可以是个人，也可以是组织。发布时选了哪个 owner，就要让包名和这个身份匹配。

个人 owner

像“这是小明自己的摊位”。别人不能随便把东西摆到小明摊位上。

组织 owner

像一个小团队共用的摊位，成员按权限一起维护。

scope 必须匹配

如果包名写着一个店名，却想发布到另一个 owner 下面，ClawHub 会拒绝。

第二站

skill 和 plugin 是两种不同盒子

skill 更像一份会教 OpenClaw 怎么做事的说明书；plugin 更像可以装进系统的能力零件。发布时，它们看的材料也不完全一样。

S

Skill

核心是一个文件夹和 SKILL.md。描述写得清楚，别人更容易知道该不该装。

P

Plugin

通常像 npm 包一样，需要清楚的包名、版本、入口和兼容信息。

V

Version

每次发布都要有版本号，像给每批货打日期和编号。

F

Files

只把该发布的文件放进包里，不把临时文件、秘密钥匙和杂物塞进去。

发布流程

一次发布像过四道小门

先收集 metadata 和文件，再把请求发给 ClawHub。服务器会检查权限、名字、版本和内容，然后才把它放到公开货架。

1. 收拾盒子

准备说明、版本、文件和入口。

2. 送到仓库

CLI、网页或工作流把发布请求交给 ClawHub。

3. 后台检查

检查 owner、scope、版本、文件和安全线索。

4. 上架

检查通过后，别人才能搜索、查看和安装。
