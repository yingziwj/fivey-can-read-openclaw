先讲人话版

📊 Usage Tracking 这页，其实是在讲“别再自己瞎估费用了，能问 provider 就直接问 provider 账本”

它主打的是实报实销式的查看，不是根据 token 硬猜一个大概数。

也就是说，能从官方 usage 接口直接拉到账单窗口，就直接拉；拉不到，就别装得很确定。

优先 provider 真实 usage / quota 重点是 /status、/usage、CLI 状态卡 查看官方原文

第一站

🧾 这套追踪分两种：一种看 provider 真账本，一种看本地会话小账本

别把这两个来源混成一个。

🏦

Provider usage

直接去供应商 usage/quota 接口拿真实窗口数据，最像官方账本。

🧮

Local cost summary

像 OpenClaw 自己记的小账，按会话日志汇总，不等于 provider 官方账单。

🙈

拿不到就隐藏

没有匹配凭据时，它宁可不显示，也不装作自己知道。

🎯

不主打估算

官方这里明显偏向“有真数据就展示真数据”，而不是乱猜成本。

第二站

📺 它会出现在好几个地方，就像同一份账单被贴在不同窗口

读者最常碰到的是聊天里的状态卡和 CLI 的汇总。

/status

像聊天窗口里的总览卡。会顺手展示当前 provider 的 usage 情况。

/usage off|tokens|full

像给每次回复屁股后面加不加“小票尾巴”。

/usage cost

这句更像看本地累计花费小账本，不是 provider 官方后台。

openclaw status --usage

像把账单打印到终端里，做一份更完整的分 provider 清单。

第三站

🔐 能不能看到 usage，关键取决于你手上有没有对应 provider 的凭据

没有钥匙，就进不了账房。

OAuth providers

像 Anthropic、Copilot、Gemini CLI 这类，通常靠 auth profile 里的 OAuth 身份去查。

API key providers

像 MiniMax、z.ai 这种，则用 API key 去问账。

当前 provider 优先

它更关心你当前正在用的模型供应商，不会一上来把全世界账本都摊一桌。

🎈 最后记一句

Usage tracking 的核心不是“估个差不多”，而是“能拿真账单就拿真账单”。
