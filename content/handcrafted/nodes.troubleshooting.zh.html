先讲人话版

🩺 Node Troubleshooting 这页，其实是在教你“节点明明在线，但工具就是不干活时，应该先查哪一层门”

它不是一句“重启试试”，而是一张从总状态到节点能力再到权限的排查梯子。

节点故障排查梯子重点是 status、describe、approvals、foreground、permissions查看官方原文

第一站

🪜 先爬总梯子，再爬节点专用梯子

别一上来就钻某个子命令，先看大门有没有开。

openclaw status

像先看整栋楼总控板，有没有明显断电。

openclaw gateway status

确认总指挥室是不是在线。

openclaw logs --follow

像站在后台玻璃窗后面看现场，到底卡在哪一步。

openclaw nodes describe --node ...

再看具体这台 node 会不会你要调用的那个能力。

第二站

🚪 Node 工具失败，最常见其实是卡在三道门：前台、权限、审批

把这三层分清，排查速度会快很多。

📱

Foreground

camera.*、canvas.* 这类常要求 app 在前台，后台直接拒。

🔐

Permissions

没给相机、屏幕录制、定位权限，再漂亮的命令也没戏。

✅

Device pairing

这是“设备能不能连到网关”的门，不是命令执行审批门。

📝

Exec approvals

这是 system.run 这种命令能不能在 node 主机上真正落地的另一道门。

第三站

🎈 最容易混的是 pairing 和 approvals，它们不是一回事

能连上节点，不等于节点就被允许帮你跑 shell。一个是进楼门卡，一个是机房操作证。
