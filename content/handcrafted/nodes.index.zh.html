先讲人话版

🛰️ Nodes 这页，其实是在讲“给 Gateway 接上一堆会动手的外设小伙伴”

Gateway 像总指挥，node 像身边的摄像头、手机、麦克风、执行机。

它们不是第二个 Gateway，而是被总指挥调度的外围设备。

外围设备，不是第二个网关重点是 pairing、status、node.invoke查看官方原文

第一站

🔌 Node 最像“插在 Gateway 身边的一只工具手臂”

消息还是先进 Gateway，再由 Gateway 决定要不要叫某个 node 出手。

🧠

Gateway

负责收消息、跑模型、发号施令。

🤖

Node

负责拍照、录音、执行本机命令、弹通知这些“动手型”工作。

📡

WebSocket

Node 通过 Gateway 的 WS 口连进来，像挂上一根长线听调度。

🧰

node.invoke

这是总指挥给 node 下动作指令的总入口。

第二站

🪪 先配对，再看状态，再决定能不能真正干活

官方把“能连上”和“被批准上岗”分开了。

openclaw devices list

像翻设备候审名单，看看谁正在门口等审批。

openclaw devices approve <requestId>

像给某台设备发正式工牌，让它能以 node 身份入场。

openclaw nodes status

像看值班板，确认哪些 node 现在在线、已配对、可调度。

openclaw nodes describe --node ...

像打开某台设备的详细档案，看它会哪些本领。

第三站

🏃 如果要在别的机器上跑命令，就把那个机器当成 node host

模型仍然只和 Gateway 说话，只是 Gateway 会把执行任务转发给远程 node 主机。

Gateway host

像总调度室，负责决定“现在该干什么”。

Node host

像远程执行车间，真正去跑 system.run 之类动作。

Approvals

执行审批是卡在 node host 上的，不是随便转过去就能跑。

🎈 最后记一句

Node 是外设，不是分身网关。它负责出手，不负责总指挥。
